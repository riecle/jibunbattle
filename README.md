# 未証明の怪物 — 構築・運用 手順書

> あなたの価値は、まだ証明されていない。でも、測定することはできる。

このリポジトリは、フィールド別に自分の現在地を観測する Web アプリ「未証明の怪物」の **Phase 0 一式**と、それを育てる**自動ループの工場**です。
プログラミングに詳しくなくても、この手順書どおりに進めれば公開できます。

---

## 0. これは何か（30秒）

ユーザーは「核刻印」（いまの自己認識）を入れ、「見せ方デッキ」を選び、**資本主義フィールド**に挑む。
返ってくるのは点数ではなく、**入力を引用した「場面」**——「君は可能性を語ったが、成果への変換は書いていない。ただしこの場が見れない構想力が滲んでいる。混沌フィールドへ行け」。
勝敗はスコアではなく観測。これを共有すると、別の人が自分でも刻みに来る。これが回ると勝手に広がる。

### この設計のいちばん大事な判断（読んでおくこと）

**実行時にAI（LLM）を呼びません。** 読みは、運営者が書いた「フィールドルーブリック」＋ユーザー入力を、ブラウザの中のルールで組み立てます。理由は3つ：

1. **バズに耐える** — 静的サイトはアクセスが何万でも無料。もしLLMを実行時に呼ぶ設計だと、バズった瞬間に課金 or 制限で落ちる。いちばん落ちてほしくない時に落ちる。
2. **正直で安全** — 入力がブラウザの外に出ない（プライバシーがクリーン）。
3. **規律** — まず一番安い形で「人は本当にシェアするのか？」を検証する。LLMでリッチにするのは、データがそれを正当化してから（Phase 1）。

LLM（Gemini）は **裏方の工場**として使います。フィールドの中身を著作・品質審査するためで、ユーザーに読みを返すためではありません。

## 1. 全体像（3つの入れ子ループ）

| ループ | 速さ | 自動化 | 中身 | どこ |
|---|---|---|---|---|
| **1. コンテンツ自己改善** | 速い | 完全自動（Phase 1） | 読みの生成と品質審査（critic が防火壁） | `factory/` |
| **2. バズ・フィードバック** | 中 | 半自動 | 何が共有されたかを計り、生成に戻す | GoatCounter + `ledger/` |
| **3. フィールド著作** | 遅い | 人間（＝あなたの堀） | 新フィールドを盲点条項込みで書く | `factory/prompts/field_author.md` |

速い2つを自動化し、遅い1つ（フィールド著作＝コピーされない価値）だけ人間に残す。これが全体の骨格です。

---

## 2. 守る不変（これは絶対に回さない芯）

何を改造してもいいが、この2つだけは変えない：

1. **スコアは飾り、観測が真実。** ランク・ティア・点数を主役にしない。読みは必ず入力を引用した「場面」。
2. **生ログは捨てない。** 観測（共有・挑戦・変遷）は積み上げる。スコアはその上に乗る、いつでも再計算できるビュー。

バズを貪欲に追ってよい。ただし**追えるのは「どの正直な場面を前に出すか」だけで、「正直をやめるか」ではない。** この線を越えたらドリフト。

## 3. Phase 0 を公開する手順

### 用意するもの
- GitHub アカウント（無料）。**Phase 0 はこれだけ。**
- （任意）GoatCounter アカウント（無料・計測用）。
- （Phase 1 で）Gemini API キー（無料枠あり）。今はいりません。

---

### 手順A — リポジトリを作る

1. GitHub にログイン → 右上「＋」→ **New repository**。
2. Repository name は自由（例：`unproven-monster`）。**Public** を選ぶ（GitHub Pages の無料公開に必要）。
3. **Create repository**。

### 手順B — ファイルを置く（2通り。好きな方）

**B-1. コマンドが使える人（git）**
```bash
# 配布zipを展開したフォルダの中で:
git init
git add .
git commit -m "init: unproven-monster phase 0"
git branch -M main
git remote add origin https://github.com/<あなた>/<リポジトリ名>.git
git push -u origin main
```

**B-2. コマンドを使わない人（GitHub のWebだけ）**
1. 作ったリポジトリの画面で **Add file → Upload files**。
2. 配布zipを展開し、**中身（`docs/` `factory/` `README.md` など）をまるごとドラッグ＆ドロップ**。
   - フォルダごとドラッグすれば階層は保たれます。`docs` フォルダはそのまま `docs` として上げること。
3. 下の **Commit changes** を押す。

### 手順C — GitHub Pages を有効化する

1. リポジトリの **Settings** → 左メニュー **Pages**。
2. **Build and deployment** の Source を **Deploy from a branch** に。
3. Branch を **`main`**、フォルダを **`/docs`** に設定 → **Save**。
4. 1〜2分待つと、同じ Pages 画面の上部に公開URL（`https://<あなた>.github.io/<リポジトリ名>/`）が出ます。開いて動けば成功。

### 手順D（任意）— 計測をつなぐ（GoatCounter）

「セッションあたり共有率」を見るために、無料の GoatCounter を繋ぎます。Phase 0 で唯一見るべき指標です。

1. https://www.goatcounter.com/ で無料サインアップ。コード（`yourname`）が決まる。
2. `docs/index.html` の最下部、この行を探す：
   ```html
   <script data-goatcounter="https://YOURCODE.goatcounter.com/count" async src="//gc.zgo.at/count.js"></script>
   ```
   `YOURCODE` を自分のコードに書き換えてコミット。
3. GoatCounter のダッシュボードで、`battle/...`（挑戦数）と `share/...`（共有数）がイベントとして見えます。
   **共有率 = share の数 ÷ battle の数。** これが主要指標。

※ 設定しなくてもアプリは普通に動きます（計測されないだけ）。

## 4. フィールドを増やす（運営者の本業 = ループ3）

Phase 0 は **資本主義フィールド1つだけ**で出します（意図的）。まず1フィールドで「シェアされるか」を見る。
当たりの兆しが出たら、次のフィールドを足す。手順：

1. `factory/prompts/field_author.md` を開き、冒頭の「フィールド名／測る価値／役割」を埋める（例：混沌フィールド）。
2. それを Claude か Gemini に丸ごと渡す。出力された JSON を `docs/content/field_<id>.json` として保存。
3. **盲点条項が入っているか**だけ必ず確認（プロンプトが強制しているが、最終チェックは人間）。
4. フィールドを切り替えるには、`docs/app.js` の先頭：
   ```js
   const FIELD_URL = "./content/field_shihonshugi.json";
   ```
   を新しいファイルに変えるだけ。
   - 複数フィールドを同時に出す「フィールド選択UI」は Phase 1 の拡張です。Phase 0 は1フィールドで十分。

> 重要な規律：**ドキュメントの中で5つフィールドを書かない。** 1つ書いて、出して、シェアされるか見て、次。フィールドの良し悪しは生の反応に当たって初めて分かる。1観測 / 1実装 / 1記録。

デッキを足したいときは `factory/prompts/deck_author.md` を同じ要領で。出力を `docs/content/decks.json` の `decks` 配列に追記。

## 5. 日々どう回すか（ループの運転）

1. **日次**：何もしなくていい。静的サイトは勝手に動く。
2. **週1**：GoatCounter を開いて**共有率**を見る。そして判断を**1個だけ**下す：
   - 共有率が伸びている → そのフィールド/デッキに燃料（露出・SNS投稿）。
   - 平板 → フィールドのルーブリックが弱い。`field_author.md` で書き直すか、別フィールドを試す。
3. **go/no-go ゲート**：1つのフィールドを十分な人数に見せて共有率が閾値を超えなければ、殺すか作り直す。超えたら伸ばす。
   - 安く撃つ（フィールド/デッキを足す）→ 共有を計る → 外れを冷酷に殺す → 当たりに燃料。これが運転の全て。

ダッシュボードを作り込まないこと。見るのは共有率ひとつ。

## 6. Phase 1 を有効化する（あとで・任意）

シェアの信号が出てきたら、裏方の工場を起動して読みの品質を自動審査します。

1. Google AI Studio で無料の Gemini API キーを取得。
2. リポジトリ **Settings → Secrets and variables → Actions → New repository secret**
   - Name: `GEMINI_API_KEY` / Value: 取得した鍵。
3. **Actions** タブ → `factory` → **Run workflow**。critic が場面を採点し、weak率が25%を超えれば改善提案を出します（自動適用はしない／あなたが読んで決める）。
4. 手元で回すなら：
   ```bash
   cd factory
   GEMINI_API_KEY=あなたの鍵 node critique.mjs shihonshugi   # 品質審査
   GEMINI_API_KEY=あなたの鍵 node generate.mjs shihonshugi 4 # 場面テンプレ案を増やす
   ```
   ※ `node` v18 以上。モデル名は既定 `gemini-2.0-flash`。変えたいときは `GEMINI_MODEL=...` で上書き。

本物の append-only 台帳に差し替える設計は `ledger/README.md` を参照。

## 7. トラブルシュート / FAQ

**Pレージが真っ白／読みが出ない**
→ ほぼ「`/docs` フォルダ指定し忘れ」か「公開直後でまだビルド中」。Settings → Pages を確認し、1〜2分待つ。

**手元でダブルクリックして開いたら動かない**
→ ローカルの `file://` だと JSON 読み込みがブロックされます（正常）。GitHub Pages 上では動きます。どうしても手元で見たいなら、フォルダ内で簡易サーバ：
```bash
cd docs && python3 -m http.server 8000   # → http://localhost:8000
```

**フォントが効いていない気がする**
→ Zen Maru Gothic は Google Fonts から読み込み。ネット接続が要ります（CDN）。

**フィールドの JSON を編集したら全部出なくなった**
→ JSON の構文ミス（カンマ・括弧）。https://jsonlint.com に貼って検査。

**共有ボタンが X を開く / 共有シートが出る**
→ スマホは OS の共有シート（Web Share API）、PCは X の投稿画面。どちらも正常。

**入力した内容はどこかに送られる？**
→ いいえ。読みの生成も変遷ログも、すべてブラウザの中だけ。サーバに送りません（だから安全）。
